# 听声辨情：基于 OpenVINO + Qwen3-ASR 的端侧客服语音智能质检系统

> Intel AI PC 端侧推理 | Qwen3-ASR + OpenVINO | 转录 · 情绪标记 · 质检报告

---

## 1. 项目介绍

**项目名：VoiceQC（听声辨情）**

一句话：上传一段客服通话录音（或直接粘贴转录文本），AI 自动完成语音转录、逐句情绪标注、问题类别识别与综合满意度评分，并生成可下载的 Markdown 质检报告；全程本地运行，通话内容不离开设备，保护客户隐私。

**完整代码**：https://github.com/leezcheng/voiceqc

**灵感**：客服中心每天产生大量通话录音，人工质检效率低、覆盖率不足 5%；大模型端侧推理使得"全量自动质检"在 AI PC 上成为可能。

```
客服录音（WAV/MP3）
       ↓  Qwen3-ASR（OpenVINO 加速）
转录文本
       ↓  情绪分析器（sentiment_analyzer.py）
逐句情绪标注  😠愤怒 / 😰焦虑 / 😊满意 / 😐中性
       ↓
质检报告：问题分类 + 解决状态 + 高风险语句 + 建议
```

## 2. 应用场景与创新点

### 2.1 面向用户

- **客服质检团队**：传统人工抽听覆盖率低，本项目可实现全量通话自动标注，聚焦高风险工单。
- **客服主管 / 运营**：实时掌握情绪分布趋势、常见投诉类别，快速发现服务短板。
- **合规 / 法务**：本地推理不出数据，满足通话内容不得上传第三方的合规要求。
- **中小企业**：无需采购昂贵的 SaaS 质检服务，在一台 AI PC 上即可部署使用。

### 2.2 解决的实际问题

1. **覆盖率**：人工质检每天只能抽听 3–5% 的录音；本方案秒级完成单条分析，支持全量处理。
2. **一致性**：人工评分主观差异大；关键词规则 + 加权评分给出可重复、可审计的标准化结果。
3. **隐私合规**：全程在端侧运行，无需将客户通话上传至云端 API，满足金融、医疗等行业的数据合规要求。
4. **轻量可扩展**：情绪分析器为纯 Python 规则引擎，无需额外 ML 模型，在 CPU 上毫秒级完成单句分析；关键词词库可按业务场景快速扩充。

### 2.3 核心创新点：规则引擎情绪分析器

本项目的核心创新在于 `sentiment_analyzer.py`，它在 Qwen3-ASR 转录输出的基础上，**不依赖任何额外 ML 模型**，通过三层机制完成情绪分析：

1. **关键词正则匹配**：100+ 个领域词条，涵盖投诉举报、退款索赔、焦虑催单、满意致谢等场景。
2. **加权情绪评分**：不同情绪关键词赋予差异化权重（愤怒 ×0.55，焦虑 ×0.30，满意 ×0.40），综合归一到 −1.0 ~ +1.0。
3. **多维度输出**：逐句情绪标签 + 7 类问题分类 + 解决状态识别 + 说话人（客服/客户）区分。

## 3. 系统总体流程（Pipeline）

整体流程分为四步（对应 `voiceqc.ipynb` 的各节与 `gradio_helper.py` 的 `run_pipeline()`）：

1. **Step A — 语音转录**：`OVQwen3ASRModel.transcribe()`，OpenVINO 加速，支持 52+ 语言
2. **Step B — 逐句情绪分析**：`sentiment_analyzer.analyze(transcript)`，纯 Python，毫秒级
3. **Step C — 报告生成**：`_build_report_md(report)`，生成结构化 Markdown
4. **Step D — 展示与下载**：Gradio 界面展示 + 一键导出 `.md`

**输出文件（Notebook 运行示例）：**

```

```

## 4. 模块设计与关键实现点

### 4.1 sentiment_analyzer.py：情绪分析引擎（核心创新）

**目标**：对 Qwen3-ASR 输出的转录文本做逐句情绪标注，生成可供质检员直接使用的结构化报告。

**关键设计：**

- **情绪优先级链**：愤怒 > 焦虑 > 轻微不满 > 满意 > 中性。同一句话同时命中愤怒和满意词条时，优先标注为愤怒（"投诉了但顺带道了谢"属于高风险）。
- **说话人启发式区分**：通过 `AGENT_MARKERS`（"您好""非常抱歉""已为您"等客服常用开场语）区分客服与客户，避免将"非常抱歉"误判为满意情绪。
- **问题类别检测**：7 类（物流/质量/退款/态度/账号/支付/商品描述），一次扫描全文即可完成分类。
- **解决状态识别**：检测"已为您处理""退款成功""24小时内专员联系"等承诺句，判断本次通话是否达成解决。

**核心数据结构：**

```python
@dataclass
class SentenceResult:
    text: str
    emotion: str        # angry / anxious / satisfied / negative_weak / neutral
    score: float        # -1.0（极负面）~ 1.0（极正面）
    keywords_hit: List[str]
    is_agent: bool      # True = 客服说话人

@dataclass
class AnalysisReport:
    sentences: List[SentenceResult]
    overall_score: float
    satisfaction_label: str   # 非常满意⭐×5 ~ 非常不满⭐×1
    issues_detected: List[str]
    resolution_detected: bool
    angry_count / anxious_count / satisfied_count / neutral_count: int
    annotated_text: str       # 可直接展示的逐句标注
    summary_md: str           # Markdown 质检报告全文
```

### 4.2 voiceqc.ipynb：端到端演示 Notebook

**关键设计：**

- **绝对路径设计**：所有 Cell 使用绝对路径，避免因 Kernel 重启导致 `WORK_DIR` 丢失的 NameError。
- **ipywidgets 兼容**：VS Code 下改用 `class _Device: value = 'CPU'` 替代下拉组件，保证 Notebook 在 VS Code 中可完整运行。
- **sys.path 隔离**：Gradio Cell 用 `importlib.util.spec_from_file_location` 精确加载本地 `gradio_helper.py`，避免与 lab2 同名模块冲突。
- **批量场景对比**：内置三种典型客服场景，运行后自动生成 matplotlib 情绪对比图。

### 4.3 gradio_helper.py：一键演示界面

**关键设计：**

- **多源音频归一化**：统一处理 Gradio 的 numpy tuple 格式与文件路径，转换为 16-bit WAV 供 ASR 使用。
- **文本 Bypass 模式**：勾选"使用文字输入"后直接跳过 ASR，便于在没有麦克风的环境下快速演示情绪分析效果。
- **三个内置示例**：强投诉、退款、支付多扣三种场景，点击示例后勾选 Bypass 即可一键查看完整质检报告。

## 8. 跑通过程与踩坑记录

### 8.1 PyTorch 版本号错误：requirements.txt 指定了不存在的版本

**现象**：`pip install -r requirements.txt` 报 `Could not find a version that satisfies the requirement torch==2.8`。

**根因**：官方 requirements.txt 写的 `torch==2.8` 版本号不存在（PyPI 上无此版本）。

**修复**：手动指定 `pip install torch==2.9.0+cpu --extra-index-url https://download.pytorch.org/whl/cpu`，实测 2.9.0+cpu 与 OpenVINO 2026.1.0 完全兼容。

### 8.2 VS Code 下 ipywidgets 渲染失败

**现象**：Cell 运行后出现 `Cannot read properties of undefined (reading 'ipywidgetsKernel')`，设备选择组件无法显示。

**根因**：VS Code Jupyter 扩展对 `ipywidgets` 的渲染存在已知兼容问题。

**修复**：将设备选择下拉改为纯 Python 类，不依赖任何 widget：

```python
class _Device:
    value = 'CPU'
device = _Device()
```

### 8.3 Kernel 重启后 NameError：变量丢失

**现象**：Kernel 重启后单独运行加载模型的 Cell，报 `NameError: name 'asr_model_dir' is not defined`。

**根因**：`asr_model_dir` 在上一个 Cell 中定义，Kernel 重启后 Cell 状态清空，依赖变量丢失。

**修复**：在每个关键 Cell 内部直接用绝对路径定义所有依赖变量，做到 Cell 完全独立、可单独重新运行。

### 8.4 模型下载路径混乱：两个目录各一份

**现象**：模型下载到 `voiceqc/Qwen3-ASR-0.6B-fp16-ov/`，但加载代码指向根目录路径，导致 `RuntimeError: Empty weights data in bin file`。

**修复**：删除所有副本，统一下载路径，在代码中硬编码绝对路径保证唯一性。

### 8.5 Gradio TypeError：同名模块冲突

**现象**：Gradio Cell 报 `TypeError: make_demo() got an unexpected keyword argument 'asr_model'. Did you mean 'ov_model'?`

**根因**：加载 ASR 模型时执行了 `sys.path.insert(0, lab2_dir)`，Python 从 `lab2-speech-recognition/gradio_helper.py` 导入了参数名为 `ov_model` 的同名函数，而非本项目的 `gradio_helper.py`。

**修复**：使用 `importlib.util.spec_from_file_location` 精确指定文件路径导入，完全绕过 `sys.path` 搜索顺序：

```python
import importlib.util
_spec = importlib.util.spec_from_file_location(
    'gradio_helper',
    r'D:\zicheng_lee\modelscope-workshop\voiceqc\gradio_helper.py'
)
_mod = importlib.util.module_from_spec(_spec)
_spec.loader.exec_module(_mod)
make_demo = _mod.make_demo
```

## 9. 总结与展望

### 9.1 端侧部署心得

- **端侧落地的主要挑战**通常不在"模型能不能跑"，而在工程细节：Python 虚拟环境的版本兼容性、同名模块的 `sys.path` 冲突、Kernel 状态与 Cell 执行顺序的耦合问题等。
- **OpenVINO 的价值**在于让 Qwen3-ASR 在纯 CPU 上稳定运行（无需 GPU），端到端延迟可控，适合在普通 AI PC 上部署生产级质检流程。
- **规则引擎 + 大模型 ASR 的组合**是本项目的核心工程判断：情绪分析不一定需要另一个大模型，领域关键词规则在客服场景准确率已经足够高，且推理延迟为毫秒级，大幅降低系统复杂度。

### 9.2 当前局限

- **规则覆盖有限**：关键词词库基于常见投诉场景构建，对新颖的委婉抱怨表达识别能力有限。
- **说话人区分为启发式**：未使用说话人分离（Speaker Diarization），对混合说话场景准确率会下降。
- **暂不支持实时流式处理**：当前为录音文件批处理模式，尚未支持通话进行中的实时质检。

### 9.3 未来优化方向

- **接入 LLM 辅助分析**：对规则引擎无法置信分类的"模糊句子"，调用本地轻量 LLM 做二次判断。
- **说话人分离**：集成 OpenVINO 加速的 Speaker Diarization 模型，更准确区分客服与客户。
- **实时流式质检**：结合流式 ASR 输出，在通话进行中实时给客服推送情绪预警提示。
- **可视化大屏**：汇总多条通话的情绪趋势、热点问题类别，生成班次级质检报告。
